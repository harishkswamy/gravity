Terminology

Component (Flyweight)

Component is a managed unit of software that performs a specific task and is part of a bigger cohesive system.

  • A component has a clearly defined interface and of course an implementation.

  • A component has a well defined lifecycle: instatiation, injection, start-up and shutdown.

  • A component is uniquely identified by the component key which is typically comprised of the component interface and an optional string identifier.

  • A component is a factory for component instances (similar to java.lang.Class).

  • A component produces component instances using a Component Strategy.

  • A component can change its strategy on the fly at runtime. When the strategy is changed, all component instances (including the ones that were created prior to the strategy change) will behave in accordance with the new strategy.

Component Strategy (Strategy, Decorator)

Component Strategy is the technique used by the component in producing instances. The strategy determines the identity of the instances produced by the component.

  • Component strategies are of two categories: Lazy Loading and Dispatching. All strategies fall under one of these two categories; Singleton and Pooling fall under Lazy Loading and Thread Local falls under Dispatching.

  • Component strategies are decoratable; a component may use Thread Local strategy, and the Thread Local strategy may in turn use a Pooling strategy.

Component Proxy (Proxy, Singleton)

Component Proxy is the component counterpart that produces component proxy instances.

  • A component proxy instance will stand-in for the component instance and control access to it; it can be thought of as a component instance controller. Hence the ability to change the behavior of the component instance when the component strategy changes.

  • A component proxy instance will be composed of the component - the instance of which is being proxied.

  • A component proxy instance will either cache the component instance and delegate all calls to the cached instance, or it will dispatch all calls to the component. The behavior is dictated by the component strategy at the time of the call.

Container (Flyweight Factory)

Container is the factory for components. Container creates and caches components based on the component key. Container is also a component manager; it manages the lifecycle of the component.

Module

A module is simply a collection of components grouped together primarily for the purpose of manageability. A module is typically a logical grouping of components.

Plugin (Plugin)

A plugin is a collection of related components grouped together in different modules. A plugin is typically an application that accomplishes a specific goal.
